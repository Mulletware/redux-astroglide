# Releasing & Publishing

There is no manual `npm publish` step. Releases are fully automated via
**[semantic-release](https://github.com/semantic-release/semantic-release)**, driven by
GitHub Actions on every push to `main`.

## TL;DR

```bash
# 1. write conventional-commit messages as you work
git commit -m "fix(exports): add proxy package.json for legacy resolvers"

# 2. push to the default branch — that's it
git push origin main
```

Pushing to `main` triggers the **Release** workflow
([`.github/workflows/release.yml`](.github/workflows/release.yml)), which builds, tests,
validates the package, and then publishes to npm if a release is warranted.

## How a version bump is decided

Versions come from your commit messages — analyzed by
`@semantic-release/commit-analyzer` using the
[Conventional Commits](https://www.conventionalcommits.org/) spec.

| Commit type                                   | Bump            | Example                         |
| --------------------------------------------- | --------------- | ------------------------------- |
| `fix(scope): ...`                             | **patch** `0.x` | `fix(exports): ...`             |
| `feat(scope): ...`                            | **minor** `x.0` | `feat: add foo plugin`          |
| `feat!:` or `BREAKING CHANGE:` in the footer | **major** `0.0` | `feat!: drop Node 14`           |
| `chore:` / `docs:` / `test:` / `style:`       | **no release**  | `docs: fix typo`                |

If the collection of commits since the last release contains nothing releasable,
semantic-release exits without publishing anything.

## What the Release workflow does

On push to `main`/`master`, [`.github/workflows/release.yml`](.github/workflows/release.yml)
runs, in order:

1. `pnpm install`
2. `pnpm build` (`rollup` + `tsc --emitDeclarationOnly`)
3. `pnpm test` (jest)
4. `pnpm validate:exports` — every `exports` entry + legacy proxy resolves
5. `pnpm test:exports` — `require`/`import` every subpath
6. `pnpm lint:package` (`publint`)
7. `npx semantic-release`

If all steps pass, semantic-release ([`.releaserc.json`](.releaserc.json)) then:

1. Determines the next version from the commits.
2. Generates release notes.
3. Updates [`CHANGELOG.md`](CHANGELOG.md).
4. Publishes to npm via `@semantic-release/npm`
   (under `publishConfig: { "access": "public" }`).
5. Commits `package.json` + `pnpm-lock.yaml` + `CHANGELOG.md` as
   `chore(release): <version> [skip ci]`.
6. Creates a GitHub release.

## Publish-time validation hooks

`npm publish` itself is guarded by lifecycle hooks in
[`package.json`](package.json), so an invalid package cannot ship even if run by hand:

- `prepack` → `npm run validate:exports && npm run lint:package`
- `prepublishOnly` → `npm run build && npm run test:exports`

The legacy-resolver proxy `package.json` files
(`plugins/*/package.json`, `selectors/package.json`) are validated by `validate:exports`
and regenerated by `sync:exports`, so they always stay consistent with the `exports` map.

## Authentication: npm granular access token

This repo publishes to npm with a **npm Granular Access Token** stored as the
`NPM_TOKEN` secret. `@semantic-release/npm` writes `${NPM_TOKEN}` into a temporary
`.npmrc` for the `npm publish` step. It's the boring, reliable, npm-documented path.

> ⚠ This is **not** npm Trusted Publishing. The tokenless OIDC path was attempted
> and is blocked on a `@semantic-release/npm` bug — see
> [Future improvement: tokenless publishing](#future-improvement-tokenless-publishing-trusted-publishing)
> below.

### One-time setup (npm-side, web UI)

Create the token once on npmjs.com, logged in as the package owner
(`trmnl_cmdr` / `dmanke@gmail.com`):

1. **Settings → Access Tokens → Create new granular access token** with:
   - Expiration: 90 days (npm requires one; rotate before it lapses)
   - Packages: **Read and write** → **Only select packages and scopes** → `redux-astroglide`
   - Organizations: **No access**
2. Copy the token immediately (shown once).

### Publishing-access setting (npm package settings)

Under **Packages → redux-astroglide → Settings → Publishing access**, keep:

> **Require two-factor authentication or a granular access token with bypass 2FA
> enabled**

This is the option that allows the token to publish non-interactively. Do **not**
select "disallow tokens" — that would block the current publish path.

### Store it in GitHub

Add the token as a repository secret (not an org secret):

- **Settings → Secrets and variables → Actions → New repository secret**
- Name: `NPM_TOKEN` (exact)
- Value: the token

### CI wiring (already in the workflow)

- `NPM_TOKEN: ${{ secrets.NPM_TOKEN }}` on the Release step env — see
  [`.github/workflows/release.yml`](.github/workflows/release.yml).
- **No `id-token: write` permission.** This is deliberate: with it granted,
  `@semantic-release/npm@13`'s OIDC branch fires and short-circuits the working
  token path (and its OIDC code discards the token → `ENEEDAUTH`).
- `permissions: contents: write` only.
- **`GITHUB_TOKEN`** — automatically provided by GitHub Actions.
- Runs on **Node 22** with **pnpm 10**.

> The CI pnpm version (10) and the committed `pnpm-lock.yaml` (lockfile v9.0) must
> match. Don't downgrade pnpm in CI — a v9.0 lockfile can't be read by pnpm 8.

### Rotation

npm requires granular tokens to expire. Before the 90-day window closes, create a
new token and update the `NPM_TOKEN` secret. No code change is needed.

### Future improvement: tokenless publishing (Trusted Publishing)

npm [Trusted Publishing](https://docs.npmjs.com/trusted-publishers) would eliminate
the long-lived token entirely — GitHub mints a short-lived OIDC token per run and npm
trusts the authorized repo + workflow. This is the better end state and npm's
recommendation. It is **not in use today** because of a bug in
`@semantic-release/npm@13.1.5` (the newest version at time of writing):

- `exchangeToken()` mints a valid OIDC token and npm accepts it.
- `oidcContextEstablished()` reduces that token to a boolean (`!!`) and **discards
  the token string**.
- `verify-auth.js` then returns early without writing anything to `.npmrc`.
- `publish.js` runs `npm publish --userconfig <empty .npmrc>` → `ENEEDAUTH`.

Worse, granting `id-token: write` makes the broken OIDC branch fire first and
short-circuit the working token path — so Trusted Publishing can't simply be turned
on alongside the token.

**When revisiting**, the migration is *not* a config flip. Options, in order of
preference:

1. **Wait for the plugin fix.** Check if a newer `@semantic-release/npm` (or a PR)
   writes the OIDC token to `.npmrc`. If fixed, grant `id-token: write`, drop
   `NPM_TOKEN`, and switch the package's Publishing-access to "disallow tokens".
2. **Bypass the plugin's publish step.** The npm CLI itself has working OIDC code
   (`@npmcli/utils/oidc.js`). Replace the `@semantic-release/npm` publish action
   with a direct `npm publish --provenance` step that relies on the CLI's exchange.
   Prototype locally with `npm publish --dry-run --provenance` against a scratch
   package first — do **not** change this blind.

Either path must be validated end-to-end in CI before the `NPM_TOKEN` secret is
removed, because `1.0.x` shipped via the token path and a half-migration can break
publishing.

### GitHub Issues are disabled in this repo

Because Issues are turned off, `.releaserc.json` configures
`@semantic-release/github` with `"failTitle": false` and `"releasedLabels": false`.
This prevents the plugin from trying to file a "release failed" Issue (which would
return HTTP 410 and crash the `fail` step, masking the real error). If you re-enable
Issues and want failure notifications, remove those two options.

Also ensure:

- The `master` and `main` branches are listed in [`branches`](.releaserc.json)
  (they are by default here).
- Local commits follow Conventional Commits — enforced by the husky `commit-msg` hook
  via commitlint.

## Sanity-checking locally

Dry-run semantic-release to see what version and notes it _would_ cut, without
publishing:

```bash
npx semantic-release --dry-run
```

## Why we don't publish by hand

A manual `npm publish` would work (the `prepublishOnly`/`prepack` hooks still run and
guard the package), but it bypasses semantic-release's version derivation, changelog
generation, and `[skip ci]` release commits — which then conflict with the next CI run.
The intended path is always: **push a conventional commit to `main` and let CI release.**

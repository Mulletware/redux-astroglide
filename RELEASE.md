# Releasing & Publishing

There is no manual `npm publish` step. Releases are fully automated via
**[semantic-release](https://github.com/semantic-release/semantic-release)**, driven by
GitHub Actions on every push to `main`.

## TL;DR

```bash
# 1. write conventional-commit messages as you work
git commit -m "fix(exports): add proxy package.json for legacy resolvers"

# 2. push to the default branch — that's it
git push origin main
```

Pushing to `main` triggers the **Release** workflow
([`.github/workflows/release.yml`](.github/workflows/release.yml)), which builds, tests,
validates the package, and then publishes to npm if a release is warranted.

## How a version bump is decided

Versions come from your commit messages — analyzed by
`@semantic-release/commit-analyzer` using the
[Conventional Commits](https://www.conventionalcommits.org/) spec.

| Commit type                                   | Bump            | Example                         |
| --------------------------------------------- | --------------- | ------------------------------- |
| `fix(scope): ...`                             | **patch** `0.x` | `fix(exports): ...`             |
| `feat(scope): ...`                            | **minor** `x.0` | `feat: add foo plugin`          |
| `feat!:` or `BREAKING CHANGE:` in the footer | **major** `0.0` | `feat!: drop Node 14`           |
| `chore:` / `docs:` / `test:` / `style:`       | **no release**  | `docs: fix typo`                |

If the collection of commits since the last release contains nothing releasable,
semantic-release exits without publishing anything.

## What the Release workflow does

On push to `main`/`master`, [`.github/workflows/release.yml`](.github/workflows/release.yml)
runs, in order:

1. `pnpm install`
2. `pnpm build` (`rollup` + `tsc --emitDeclarationOnly`)
3. `pnpm test` (jest)
4. `pnpm validate:exports` — every `exports` entry + legacy proxy resolves
5. `pnpm test:exports` — `require`/`import` every subpath
6. `pnpm lint:package` (`publint`)
7. `npx semantic-release`

If all steps pass, semantic-release ([`.releaserc.json`](.releaserc.json)) then:

1. Determines the next version from the commits.
2. Generates release notes.
3. Updates [`CHANGELOG.md`](CHANGELOG.md).
4. Publishes to npm via `@semantic-release/npm`
   (under `publishConfig: { "access": "public" }`).
5. Commits `package.json` + `pnpm-lock.yaml` + `CHANGELOG.md` as
   `chore(release): <version> [skip ci]`.
6. Creates a GitHub release.

## Publish-time validation hooks

`npm publish` itself is guarded by lifecycle hooks in
[`package.json`](package.json), so an invalid package cannot ship even if run by hand:

- `prepack` → `npm run validate:exports && npm run lint:package`
- `prepublishOnly` → `npm run build && npm run test:exports`

The legacy-resolver proxy `package.json` files
(`plugins/*/package.json`, `selectors/package.json`) are validated by `validate:exports`
and regenerated by `sync:exports`, so they always stay consistent with the `exports` map.

## Prerequisites (one-time repo setup)

These live in the GitHub repo UI, not in code:

- **`NPM_TOKEN`** *(required — this is the current blocker)* — an npm **Granular Access
  Token** scoped to publish `redux-astroglide`. Create one at
  https://www.npmjs.com/settings/**your-account**/tokens → **Create new granular access
  token** with:
  - Expiration: 90 days (npm requires one; rotate before it lapses)
  - Packages: **Read and write** → **Only select packages and scopes** → `redux-astroglide`
  - Organizations: **No access**

  Copy the token immediately (shown once), then add it under
  **Settings → Secrets and variables → Actions → New repository secret**, named exactly
  `NPM_TOKEN`. The workflow passes it to semantic-release's `verifyConditions` step;
  without it, the Release job fails immediately with `ENONPMTOKEN`. A granular token
  satisfies publish 2FA on its own, so it won't be blocked by a 2FA prompt in CI.
- **`GITHUB_TOKEN`** — automatically provided by GitHub Actions; no setup needed.

> No `.npmrc` is committed by design — `@semantic-release/npm` reads `NPM_TOKEN` straight
> from the environment. Don't commit a `.npmrc` with a token in it.

### GitHub Issues are disabled in this repo

Because Issues are turned off, `.releaserc.json` configures
`@semantic-release/github` with `"failTitle": false` and `"releasedLabels": false`.
This prevents the plugin from trying to file a "release failed" Issue (which would
return HTTP 410 and crash the `fail` step, masking the real error). If you re-enable
Issues and want failure notifications, remove those two options.

Also ensure:

- The `master` and `main` branches are listed in [`branches`](.releaserc.json)
  (they are by default here).
- Local commits follow Conventional Commits — enforced by the husky `commit-msg` hook
  via commitlint.

## Sanity-checking locally

Dry-run semantic-release to see what version and notes it _would_ cut, without
publishing:

```bash
npx semantic-release --dry-run
```

## Why we don't publish by hand

A manual `npm publish` would work (the `prepublishOnly`/`prepack` hooks still run and
guard the package), but it bypasses semantic-release's version derivation, changelog
generation, and `[skip ci]` release commits — which then conflict with the next CI run.
The intended path is always: **push a conventional commit to `main` and let CI release.**
